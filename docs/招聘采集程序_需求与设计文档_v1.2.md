## 0. 文档目的与范围

本文档定义一个独立的招聘信息采集程序（下称"采集器"）的需求与技术设计。采集器从四个西班牙华人信息门户采集招聘信息，经清洗、三级去重后，作为 zhaopin.es 冷启动期的填充数据与去重参考。

采集目标站点：欧浪网(https://infohuaxin.com/showclass.asp?class1=13)、欧华网、华人街、西华网。

**核心边界（务必贯穿全程）：**

- 采集器**物理独立**于 zhaopin.es 主站，拥有独立代码库与独立数据库（`crawler_db`），仅以只读方式访问主库做去重比对，不侵入主站业务逻辑。
- 采集数据为**临时性质**。冷启动完成后，这批数据即视为过期，提供一键清理机制，采集模块可整体下线。
- 采集须遵守采集礼仪（频率控制、错峰、尊重 robots），避免对目标站造成压力，也避免自身被封。

## 1. 项目定位与合规边界

### 1.1 定位

采集数据仅用于 zhaopin.es 冷启动阶段的内容填充与市场参考。定位为"临时脚手架"，非长期核心资产。上线成熟后，自有 UGC（用户发布内容）取代采集数据，采集数据清除。

### 1.2 合规风险与应对

尽管定位为临时使用，仍存在以下风险，需在实现与运营中主动控制：

| 风险 | 说明 | 应对 |
|---|---|---|
| 著作权 | 招聘文案多为发布者原创，批量搬运展示可能侵权 | 冷启动期用完即删；不长期留存；优先展示结构化字段（岗位/薪资/地区）而非原文大段照搬；保留来源标记便于溯源与删除 |
| 服务条款 / robots | 目标站 ToS 可能禁止抓取 | 采集前检查各站 robots.txt；控制频率，模拟正常访问节奏；不绕过登录墙抓取受保护内容 |
| GDPR / 个人数据 | 联系方式含个人数据（电话、微信、姓名） | 最小化采集，仅取招聘必需字段；不长期留存；清理机制到位；不将采集来的个人数据用于招聘用途以外的任何场景 |
| 品牌风险 | 西班牙华人圈小，被同行发现"搬运"影响 zhaopin 口碑 | 低调采集；数据仅作冷启动过渡；成熟后彻底替换 |

> **实现层建议：** 采集库中每条记录都必须带 `source_site`、`source_url`、`collected_at`、`purge_after`（预期清理日期）字段，使"用完即删"在数据层面可执行、可审计。

## 3. 去重设计（核心）

去重是本程序的技术核心。分三级，全部启用。

### 3.1 三级去重总览

```
采集到新记录
   │
   ├─ 一级：采集库内 URL 去重（同站重复采集）
   │        source_url 已存在 → 丢弃
   │
   ├─ 二级：跨站内容去重（同一招聘在多站重发）
   │        联系方式归一化键命中 → 视为重复
   │        或 内容指纹(SimHash)相似度超阈值 → 视为重复
   │
   └─ 三级：与 zhaopin 主库去重（防止导入已有岗位）
            联系方式键 / 指纹 命中主库 → 标记为"主库已存在"，不导入
```

### 3.2 一级：采集库内 URL 去重

最简单、最可靠。`source_url` 做唯一索引，采集前先查库，已存在则跳过。同时用于增量采集（见 §6.4）。

### 3.3 二级 & 三级：内容指纹与联系方式归一化

同一招聘常在多站重复发布，URL 不同，无法靠 URL 去重。采用两个信号：

**信号 A — 联系方式归一化键（最强）**

电话和微信是去重的黄金主键。归一化处理：

- **电话：** 去除空格、连字符、括号、国际区号前缀（+34 / 0034 统一处理），提取纯数字末 9 位（西班牙手机号 9 位）。生成 `phone_norm`。
- **微信：** 转小写，去空格。生成 `wechat_norm`。
- 生成 `contact_key = phone_norm | wechat_norm`（任一命中即视为潜在同源）。

同一 `contact_key` + 相近发布时间 + 相似岗位 → 高置信度判定为重复。

**信号 B — 内容指纹 SimHash（用于文案微调场景）**

发布者常改动文案（改薪资、加一句话）后重发，联系方式相同但也可能因换号而不同。对 `title + company + description` 归一化文本计算 **SimHash（64 位）**，比较汉明距离：

- 距离 ≤ 3：判定重复
- 距离 4–8：进入**人工复核队列**（`cj_review_queue`）
- 距离 > 8：判定为不同岗位

> SimHash 适合中文短文本且计算轻量，PHP 可用现成库或自行实现（分词 → 加权 → 按位签名）。中文分词建议引入轻量分词（如 scws / jieba-php），或退化为 2-gram 字符切分（无需分词依赖，冷启动够用）。

### 3.4 联系方式缺失时的降级去重

当联系方式无法获取（登录墙、页面未提供）：

- 去重仅依赖 **信号 B（SimHash）** + 组合键 `(title_norm, company_norm, city)`。
- 汉明距离阈值收紧（≤ 2 才自动判重），其余全部进 `cj_review_queue`，标注 `confidence = low`。
- 目的：宁可漏判（进复核）也不误判（错误合并两个不同岗位）。

### 3.5 去重判定的置信度与人工复核

每次判重写入 `cj_dedup_log`，记录命中信号、相似度、置信度。低置信度记录进入 `cj_review_queue`，冷启动期人工快速扫一遍即可（数据量在冷启动阶段有限，人工成本可控）。

## 4. 技术架构

### 4.1 技术栈

沿用 zhaopin.es 主栈，降低维护成本：

- **语言：** PHP 8.x（CLI 模式运行采集任务，非 Web）
- **数据库：** MySQL 8.x（独立库 `crawler_db`）
- **HTTP 抓取：** Guzzle（cURL 封装）
- **HTML 解析：** Symfony DomCrawler + CSS Selector，或 `paquettg/php-html-parser`
- **调度：** Linux cron
- **中文分词（可选）：** scws / jieba-php；或退化为字符 2-gram

> **JS 渲染例外：** 若某站列表/详情页为前端 JS 动态渲染（PHP 抓取拿不到内容），需引入 headless 浏览器（Puppeteer / Playwright，Node 侧），作为该站专用采集通道，输出统一 JSON 交回 PHP 处理。采集前需逐站确认是否存在此情况。

### 4.2 目录结构（公开 / 私有分离，独立于 zhaopin）

采集器采用与 zhaopin.es 相似的"公开目录 / 私有目录"分离布局，但是**一套完全独立的部署**（独立代码库、独立数据库 `crawler_db`、独立域名或子域），与主站互不共用代码。

分离原则：**只有必须经 Web 访问的文件放在 `cj_html`（web root，网络可见）；所有应用逻辑、配置、采集器、密钥、日志放在 `app`（web root 之外，网络不可见）。** 这样即使采集器挂了个内部管理页在公网，核心逻辑与数据也不会被直接下载或探测。

```
cj_project/                         # 采集器项目根（独立部署，非 zhaopin 子目录）
│
├── cj_html/                        # ← web root，唯一对外暴露的目录（网络可见）
│   ├── index.php                   # 极简内部入口（如复核/看板页），仅做请求转发到 app
│   ├── review.php                  # 人工复核队列页面入口（内部访问，建议加鉴权/IP 白名单）
│   ├── dashboard.php               # 采集运行看板入口（读 cj_crawl_runs）
│   ├── assets/                     # css / js / 图标等静态资源
│   └── .htaccess                   # 目录保护、伪静态（如用 Apache）
│
└── app/                            # ← web root 之外，网络不可访问（私有）
    ├── config/
    │   ├── config.php              # 数据库连接、密钥、Brevo 等（严禁进 web root）
    │   └── sites/                  # 每站一个采集配置（选择器、频率、字段映射）
    ├── src/
    │   ├── Fetcher/                # HTTP 抓取（频控、重试、代理、UA）
    │   ├── Parser/                 # 各站解析器 → 统一数据模型
    │   ├── Normalizer/             # 电话/微信/文本归一化
    │   ├── Dedup/                  # 三级去重引擎（URL / contact_key / SimHash）
    │   ├── Repository/             # crawler_db 读写 + zhaopin 主库只读比对
    │   ├── Scheduler/              # 任务编排、增量采集
    │   ├── Import/                 # 导入主库 + 写 cj_import_map
    │   └── Purge/                  # 一键清理（冷启动结束后）
    ├── bin/                        # CLI 入口，由 cron 调用（不经 Web）
    │   ├── crawl.php               # 采集入口（按站/全量）
    │   ├── dedup.php               # 独立去重批处理（可选）
    │   ├── import.php              # 复核通过后导入主库
    │   └── purge.php               # 数据清理
    ├── vendor/                     # Composer 依赖（Guzzle、DomCrawler 等）
    └── logs/                       # 采集日志、错误日志（不可对外）
```

**要点：**

- `cj_html` 是唯一的 web root，Web 服务器（Apache/Nginx）的 DocumentRoot 指向它。`app` 在其之上一级，Web 无法直达。
- `cj_html` 里的 PHP 入口文件保持极薄，只负责引入 `../app/` 下的逻辑并转发，不写业务代码。例如 `require __DIR__ . '/../app/bootstrap.php';`。
- 采集、去重、导入、清理这些**核心动作全部走 `app/bin/` 下的 CLI 脚本，由 cron 触发，根本不经过 Web**——网络上无从触发，也无从探测。
- `app/config/config.php` 存数据库密码、主库只读账号、Brevo 密钥等敏感信息，务必在 web root 之外。
- 内部页面（复核、看板）虽放在 `cj_html`，仍建议加基础鉴权或 IP 白名单，因为它们能看到采集数据。
- 这套布局与 zhaopin.es 同构（便于你维护时心智一致），但**独立部署**：独立库、独立配置、独立 cron，冷启动结束后整个 `cj_project` 目录连同 `crawler_db` 一并下线删除，不牵动主站。

### 4.3 site_config 结构（每站一份）

```php
return [
    'site'            => 'oulang',        // 站点标识
    'enabled'        => true,
    'list_url'        => 'https://…/jobs?page=%d',
    'list_selector'   => '.job-item a.title',   // 列表页 → 详情链接
    'detail'          => [
        'title'    => '.job-title',
        'company'  => '.company',
        'salary'   => '.salary',
        'desc'     => '.job-desc',
        'phone'    => '.contact-phone',   // 明文时
        'wechat'   => '.contact-wechat',
        'city'     => '.location',
        'date'     => '.publish-date',
    ],
    'contact_mode'    => 'plain',          // plain | click | login_wall
    'rate_limit'      => ['min_delay' => 8, 'max_delay' => 20], // 秒
    'render'          => 'php',            // php | headless

```

站点改版时只改配置，不动核心代码，这是采集器长期可维护的关键。

### 4.4 与 zhaopin 主库的比对接口

采集器**不直接写主库**，仅只读比对。两种实现，任选：

- **方案 A（推荐，解耦彻底）：** zhaopin 主站暴露一个内部去重查询接口（如 `POST /internal/dedup-check`，输入 `contact_key` / `simhash`，返回是否已存在）。采集器通过 HTTP 调用。主站不开放数据库给外部模块。
- **方案 B（简单，耦合稍紧）：** 采集器持只读账号直连主库，查询招聘表的 `contact_key` 与 `simhash` 索引列。要求主库招聘表预先存好这两列的归一化值与索引。

> 无论哪种方案，**主库招聘表都应新增 `contact_key`、`simhash` 两列并建索引**，否则三级去重无法高效执行。这是需要在 zhaopin 主库 DDL 里配合的一处改动。

### 4.5 数据流向

```
目标站 → Fetcher → Parser → Normalizer → Dedup(三级) → crawler_db.cj_jobs_clean
                                              │
                                              ├─ 命中采集库/跨站重复 → 丢弃 + cj_dedup_log
                                              ├─ 命中主库          → 标记 exists_in_main，不导入
                                              ├─ 低置信度          → cj_review_queue
                                              └─ 唯一新记录        → 待导入（import_ready=1）

冷启动导入：人工确认后，将 import_ready=1 的记录批量导入 zhaopin 主库
             每导入一条 → 在 cj_import_map 记录 (crawler_job_id ↔ main_job_id, batch)
             主库对应记录打 origin='crawler' 标记
冷启动结束：依 cj_import_map 精准清理主库对应记录 → 清空 crawler_db → 采集模块下线
```

导入主库这一步**保持人工把关**，不做全自动，避免脏数据直接进主站。导入的同时写入 `cj_import_map`（见 §5.6），建立采集库与主库的一一对应账本，为日后精准清理留好依据。

## 5. 数据库设计（crawler_db）

> **命名约定：** 采集库 `crawler_db` 中所有数据表统一使用 `cj_` 前缀（cj = 采集），与 zhaopin.es 主库表清晰区分。即便日后某些表因调试需要临时建在同一 MySQL 实例，`cj_` 前缀也能一眼辨明归属，避免误操作主库表。

### 5.1 cj_raw_pages — 原始抓取存档

```sql
CREATE TABLE cj_raw_pages (
    id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    source_site   VARCHAR(20)  NOT NULL,
    source_url    VARCHAR(768) NOT NULL,
    raw_html      MEDIUMTEXT,
    http_status   SMALLINT,
    fetched_at    DATETIME     NOT NULL,
    UNIQUE KEY uk_url (source_url)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

保留原始 HTML 便于解析器出错时重跑，不必重新抓取（也减少对目标站的请求）。

### 5.2 cj_jobs_clean — 清洗后的统一模型

```sql
CREATE TABLE cj_jobs_clean (
    id             BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    source_site    VARCHAR(20)  NOT NULL,
    source_url     VARCHAR(768) NOT NULL,
    title          VARCHAR(255),
    company        VARCHAR(255),
    category       VARCHAR(50),
    city           VARCHAR(50),
    district       VARCHAR(50),
    salary_raw     VARCHAR(100),
    description    TEXT,
    contact_phone  VARCHAR(30),
    contact_wechat VARCHAR(50),
    contact_name   VARCHAR(50),
    phone_norm     VARCHAR(15),
    wechat_norm    VARCHAR(50),
    contact_key    VARCHAR(70),          -- phone_norm|wechat_norm
    simhash        BIGINT UNSIGNED,      -- 64-bit 内容指纹
    publish_date   DATE,
    collected_at   DATETIME     NOT NULL,
    purge_after    DATE,                 -- 预期清理日期
    dedup_status   ENUM('unique','dup_site','dup_cross','exists_in_main','review') DEFAULT 'unique',
    confidence     ENUM('high','low') DEFAULT 'high',
    import_ready   TINYINT(1)   DEFAULT 0,
    imported_at    DATETIME     NULL,
    UNIQUE KEY uk_url (source_url),
    KEY idx_contact_key (contact_key),
    KEY idx_simhash (simhash),
    KEY idx_status (dedup_status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 5.3 cj_dedup_log — 去重判定日志

```sql
CREATE TABLE cj_dedup_log (
    id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    job_id        BIGINT UNSIGNED NOT NULL,
    matched_against ENUM('crawler','main') NOT NULL,
    matched_id    BIGINT UNSIGNED,       -- 命中的记录 id（采集库或主库）
    signal        ENUM('url','contact_key','simhash') NOT NULL,
    hamming_dist  TINYINT,               -- simhash 命中时的汉明距离
    decision      ENUM('dup','review','unique') NOT NULL,
    created_at    DATETIME NOT NULL,
    KEY idx_job (job_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 5.6 cj_import_map — 导入映射表（采集数据 ↔ 主库数据一一对应）

记录每条采集数据导入 zhaopin 主库后与主库记录的对应关系。**此映射只存在于采集库，主库不持有反向指针**，以便采集模块下线、采集库删除后，主库不会残留悬空引用。清理时以本表为"账本"精准定位主库对应记录。

```sql
CREATE TABLE cj_import_map (
    id             BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    crawler_job_id BIGINT UNSIGNED NOT NULL,   -- 采集库 cj_jobs_clean.id
    main_job_id    BIGINT UNSIGNED NOT NULL,   -- zhaopin 主库招聘表 id
    import_batch   VARCHAR(40)     NOT NULL,   -- 导入批次号，便于按批回滚/清理
    imported_at    DATETIME        NOT NULL,
    purged         TINYINT(1)      DEFAULT 0,  -- 主库对应记录是否已清理
    purged_at      DATETIME        NULL,
    UNIQUE KEY uk_main (main_job_id),          -- 一一对应，防止重复导入
    KEY idx_crawler (crawler_job_id),
    KEY idx_batch (import_batch)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**设计说明：**

- 唯一约束 `uk_main` 保证每条主库记录至多对应一条采集记录，防止重复导入。
- `import_batch` 支持按批次审计与回滚——某次冷启动导入若发现质量问题，可整批定位、整批清理。
- 主库侧仅需一个轻量标记 `origin`（值 `crawler`），作为清理时的双保险校验，不存指向采集库的 id。主库因此保持干净，且不依赖采集库的存续。
- 独立成表（而非在 `cj_jobs_clean` 加列）的理由：导入是一次性动作、可能一条采集数据合并到主库某条已有记录、也可能分批重导，独立映射表比在业务表堆字段更清晰、更易审计。

## 6. 反爬应对与采集礼仪

采集礼仪不仅是道德问题，也是避免被封 IP、保证采集稳定的实操要求。

### 6.1 频率控制

- 每站请求间隔随机化（如 8–20 秒），避免固定节奏被识别。
- 单站单次采集设上限（如每次最多 N 页 / N 条），分多天完成全量，而非一次抓空。
- 错峰采集，避开目标站高峰时段。

### 6.2 请求伪装（正常访问，非攻击性）

- 使用真实浏览器 UA，可轮换。
- 携带合理 Referer、Accept 头。
- 保持会话（cookie）连续，模拟正常浏览。

> 目的是"像一个正常访客"，不是对抗式绕过。若某站明确用登录墙保护内容，**不绕过**——直接跳过该内容。

### 6.3 代理（按需）

- 若单 IP 被限流，可引入少量住宅代理轮换。冷启动数据量不大，通常不需要，优先靠降低频率解决。

### 6.4 增量采集

- 每站维护已采 `source_url` 集合。后续采集只处理新出现的 URL，遇到连续 M 条已存在的记录即停止翻页（列表页按时间倒序时有效），大幅减少请求量。

### 6.5 站点改版容错

- 解析器取不到关键字段（title 连续为空）时，判定为可能改版，记录告警并暂停该站采集，避免存入大量空记录。见 §7。

## 7. 调度与监控

- **调度：** cron 按站分时段触发 `bin/crawl.php --site=xxx`，各站错开，避免同时打满带宽。
- **失败重试：** 单页抓取失败重试 2–3 次（指数退避）；整站失败记入 `cj_crawl_runs.status=failed`。
- **结构变更告警：** 某站解析成功率骤降（如 title 空值率 > 50%）→ 发告警（邮件用你现成的 Brevo），人工检查选择器。
- **运行看板（可选）：** 一个极简内部页读 `cj_crawl_runs` 展示各站最近采集量、去重率、错误数。冷启动内部用，不必精美。

## 8. 冷启动数据清理（用完即删）

这是本项目定位的关键落地环节，必须实现，不能只停留在文档：

- `purge.php` 支持：清空全部采集数据 / 按 `purge_after` 到期清理 / 按站清理 / **按 `import_batch` 清理**。
- **精准清理主库（核心）：** 以采集库 `cj_import_map` 为账本，遍历（或按批次筛选）`main_job_id`，定位主库对应记录执行下线/删除，回写 `cj_import_map.purged=1`、`purged_at`。主库侧 `origin='crawler'` 作为双保险校验，防止误删自有 UGC。
- 主库对应记录清理完成后，再清空 `crawler_db`、摘除 cron、归档下线采集模块。**顺序很重要**：必须先据 `cj_import_map` 处理完主库，再删采集库——否则账本先没了，主库的采集数据就失去精准定位依据，只能靠 `origin` 标记粗粒度清理。
- 冷启动完成、自有 UGC 达到规模后执行上述流程。若希望采集数据平滑退场（而非硬删），也可按 `import_batch` 分批下线，边替换边清理。

## 11. zhaopin.es 主库配合改动清单（DDL 补丁）

采集器独立运行，但三级去重的"与主库去重"和导入后的精准清理，需要 zhaopin.es 主库配合两处轻量改动。以下为汇总清单，便于在主库单独执行，无需回翻本文档其他章节。字段名请按主库招聘表实际命名调整。

**改动一：招聘表增加去重列（用于三级去重比对，见 §3、§4.4）**

```sql
-- 假设主库招聘表为 jobs，字段名按实际调整
ALTER TABLE jobs
    ADD COLUMN contact_key VARCHAR(70)     NULL COMMENT '电话/微信归一化去重键 phone_norm|wechat_norm',
    ADD COLUMN simhash     BIGINT UNSIGNED NULL COMMENT '内容指纹 SimHash(64bit)',
    ADD KEY idx_contact_key (contact_key),
    ADD KEY idx_simhash (simhash);
```

> 存量数据需回填一次 `contact_key` / `simhash`（跑一个一次性脚本，对已有招聘记录计算归一化键与指纹），否则采集器只能与"改动后新增"的主库记录去重。新发布的招聘应在写入时同步生成这两列。

**改动二：招聘表增加来源标记（用于清理时双保险校验，见 §5.6、§8）**

```sql
ALTER TABLE jobs
    ADD COLUMN origin VARCHAR(20) NOT NULL DEFAULT 'user'
        COMMENT '数据来源：user=自有UGC / crawler=冷启动采集导入',
    ADD KEY idx_origin (origin);
```

> 采集数据导入主库时置 `origin='crawler'`；自有用户发布保持 `origin='user'`。清理时以采集库 `cj_import_map` 为账本精准定位，`origin='crawler'` 作双保险，防止误删自有 UGC。

**不需要在主库做的事：**

- 主库**不**存指向采集库的任何 id 或外键。采集库↔主库的一一对应关系全部存于采集库 `cj_import_map`（§5.6）。这样采集模块下线、采集库删除后，主库不残留任何悬空引用。
